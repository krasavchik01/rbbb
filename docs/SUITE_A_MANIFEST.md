# Suite-A Manifest

> Живой манифест продукта: зачем существует Suite-A, какие у него принципы, текущее состояние, идеи улучшений и правила качества. Этот документ должен обновляться после крупных решений, QA-заходов и изменений продукта.

## 1. Коротко: что такое Suite-A

**Suite-A** — внутренняя операционная система компании для управления проектами, ролями, задачами, часами, утверждениями, бонусами и прозрачностью работы команды.

Продукт должен отвечать на главный вопрос:

> Кто что делает, зачем делает, сколько времени потрачено, кто за это отвечает, кто утвердил, где деньги/бонусы/статусы, и можно ли этому доверять?

Suite-A не должен быть просто набором страниц. Это должен быть **единый центр управления компанией**, где каждый участник видит ровно то, что ему нужно, и каждое важное действие оставляет понятный след.

## 2. Главная цель продукта

Сделать управление проектами компании:

- **точным** — данные не должны врать, молча пропадать или показывать нули вместо ошибок;
- **прозрачным** — видно, кто принял решение, когда, почему и что изменилось;
- **быстрым** — действия вроде добавления часов не должны заставлять ждать лишние перезагрузки;
- **ролевым** — каждая роль видит свои задачи, права и ответственность;
- **надёжным** — кнопки не должны вести на 404, пустые id или фейковый успех;
- **производственным** — система уже используется, поэтому изменения должны быть аккуратными и проверяемыми.

## 3. Для кого продукт

Основные роли:

- **CEO / admin** — полный контроль, финальные утверждения, бонусы, системная прозрачность.
- **Заместитель директора** — утверждение проектов, назначение команды, контроль загрузки.
- **Партнёр** — утверждение часов, контроль работы по проектам.
- **Project Manager / Supervisor / Assistant** — выполнение и управление задачами по проектам.
- **Procurement** — создание procurement-проектов, договоры, суммы, НДС, отклонения/доработки.
- **HR / Accountant / Admin Staff / Company Director / Contractor** — специализированные роли с ограниченным доступом.
- **Сотрудник** — видит свою работу, задачи, часы, статусы и личную часть данных.

## 4. Принципы продукта

### 4.1. Никакого фейкового успеха

Если база не сохранила действие, UI не должен говорить “успешно”.

Примеры:

- утверждение часов не прошло → показываем ошибку;
- отклонение проекта не сохранилось → проект остаётся на месте;
- загрузка часов упала → не показываем молча `0`, а логируем/показываем предупреждение.

### 4.2. Никаких пустых переходов

Кнопки и ссылки не должны вести на:

- `/404`
- `/project/undefined`
- несуществующий route
- страницу без нужного id

Каждый внутренний переход должен быть покрыт проверкой или guard-логикой.

### 4.3. Роль = права + ответственность

Если роль видит кнопку, то:

1. она должна иметь право выполнить действие;
2. backend/frontend должны это проверять;
3. действие должно быть понятно пользователю;
4. результат должен быть проверяемым.

### 4.4. Прозрачность важнее красоты

Красивый UI бесполезен, если он скрывает ошибки. Важные зоны должны показывать:

- статус загрузки;
- причину ошибки;
- кто утвердил/отклонил;
- дату и время действия;
- что именно изменилось.

### 4.5. Production safety

Suite-A содержит реальные данные компании. Поэтому:

- не трогать production-базу без явной необходимости;
- не коммитить секреты;
- не делать рискованные миграции без плана отката;
- каждое исправление проверять тестами/сборкой;
- лучше маленькие безопасные коммиты, чем огромный опасный рефакторинг.

## 6. Что сейчас считается главным риском

### P0 — критично

- Любая кнопка, которая говорит “успешно”, но данные не сохранились.
- Любой approve/reject без проверки прав.
- Любой route/action, который ведёт в 404 или `undefined`.
- Любая финансовая формула, которая может дать `NaN`, неправильную сумму или скрыть ошибку.
- Любой UI, который молча показывает нули вместо ошибки загрузки данных.

### P1 — важно

- Неполная прозрачность статусов: непонятно, кто/когда/почему изменил проект.
- Медленные экраны из-за лишних запросов или полного reload после маленького действия.
- Разные страницы по-разному трактуют один и тот же статус проекта.
- Роль видит пункт меню, но не может нормально завершить сценарий.

### P2 — улучшения

- Улучшение UX, текста ошибок, подсказок и empty states.
- Больше автотестов на конкретные клики и бизнес-сценарии.
- Единая дизайн-система статусов, бейджей, уведомлений.

## 7. Северная звезда: каким Suite-A должен стать

Suite-A должен стать системой, где:

- у каждого проекта есть понятный жизненный цикл;
- каждое действие имеет владельца;
- каждый статус объясним;
- каждый сотрудник понимает, что от него ждут;
- руководство видит реальную картину без ручного сбора информации;
- бонусы и часы считаются честно и проверяемо;
- ошибки не прячутся;
- продукт ощущается быстрым, строгим и профессиональным.

## 8. Основные продуктовые направления

### 8.1. Project Lifecycle

Цель: сделать путь проекта от создания до закрытия прозрачным.

Нужные состояния:

- создан procurement;
- ожидает утверждения;
- команда назначена;
- активный проект;
- часы собираются;
- часы утверждаются;
- проект завершён;
- бонусы рассчитаны;
- бонусы утверждены/выплачены;
- проект закрыт.

Идея улучшения: сделать отдельную визуальную timeline/историю проекта.

### 8.2. Timesheets

Цель: часы должны быть быстрыми, точными и утверждаемыми.

Важно:

- сотрудник добавляет часы быстро;
- партнёр видит pending часы;
- approve/reject работает прозрачно;
- rejected часы имеют причину;
- bonuses используют только достоверные часы;
- нет молчаливых нулей при ошибке загрузки.

### 8.3. Approvals

Цель: любое утверждение — это управляемый workflow, а не просто кнопка.

Для каждого approval нужно фиксировать:

- кто утвердил;
- когда;
- что именно было утверждено;
- какая роль имела право;
- если отклонено — почему.

### 8.4. Bonuses

Цель: бонусы должны быть честными, объяснимыми и защищёнными.

Важно:

- CEO/admin видит полную картину;
- сотрудник видит только свою часть;
- расчёты не зависят от сломанных/неподтверждённых часов;
- если часы не загрузились — это видно;
- история изменений бонусов должна быть доступна.

### 8.5. Role-based UI

Цель: каждая роль получает чистый и понятный интерфейс.

Нужно:

- пройти все роли;
- проверить sidebar;
- проверить dashboard;
- проверить основные действия;
- убрать лишнее;
- добавить недостающие подсказки.

### 8.6. Data Transparency

Цель: пользователь должен понимать состояние данных.

Нужно:

- loading states;
- error states;
- empty states;
- audit logs;
- visible sync status для важных операций.

## 9. Рабочий процесс улучшения продукта

Каждый новый QA-заход должен идти так:

1. Сформулировать область проверки.
2. Запустить агентов-тестировщиков по разным направлениям.
3. Свести находки в P0/P1/P2.
4. Исправить P0/P1 маленькими безопасными коммитами.
5. Прогнать verification bundle.
6. Обновить этот манифест, если изменилось понимание продукта.
7. Отчитаться простым языком.

## 11. Идеи для ближайших улучшений

### Ближайшие QA-заходы

- Полный role/sidebar audit: каждая роль × каждый пункт меню × каждый route.
- E2E-click smoke tests: dashboard buttons, approve/reject, project open, timesheet add.
- Approval audit log: единая история утверждений/отклонений.
- Error-state audit: найти все `.catch(() => {})` и заменить на видимые/логируемые ошибки.
- Financial calculation audit: суммы, НДС, бонусы, проценты, округления.
- Status consistency audit: единая модель статусов проекта во всех страницах.

### Продуктовые идеи

- “Центр прозрачности проекта”: timeline, решения, часы, команда, деньги, бонусы.
- “Что требует моего внимания” для каждой роли.
- “Почему я это вижу?” подсказки для ролевого доступа.
- “Проверка здоровья системы” для админа: сломанные маршруты, ошибки загрузки, pending approvals.
- “AI QA режим”: агенты регулярно проходят продукт и пишут отчёт.

## 12. Правила для AI-агентов, работающих над Suite-A

Любой агент должен помнить:

- это production-sensitive проект;
- секреты не выводить и не коммитить;
- без необходимости не трогать production-базу;
- сначала понять workflow, потом чинить;
- не считать UI-успехом то, что не подтверждено базой;
- после изменений запускать проверки;
- писать маленькие понятные коммиты;
- обновлять манифест при важных продуктовых решениях.
